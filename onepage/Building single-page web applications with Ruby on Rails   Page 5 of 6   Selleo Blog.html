Ember.js

Ember.js is the latest name for the javascript MVC framework that was previously known as Amber.js and Sproutcore 2.0. One of the most prominent creators / developers of Ember.js is Yehuda Katz, well-known for contributing significantly to the Ruby on Rails core. The Ember.js has its roots in the original Sproutcore, from which it adopted the idea of how to abstract the complexity of the observer pattern using bindings and computed properties. What is lacking in Sproutcore is a good templating solution for the view layer. Ember.js is an attempt to take the best of Sproutcore frameworks, e.g. bindings and computed properties, and to combine them with the features that cut down the boilerplate code needed to keep the views up to date. So while SproutCore 1.x is primarily a widget-centric focused framework, Ember.js aims to be a framework for building web-based applications relying on HTML and CSS for the presentation layer.

Ember.js entered the list because:

  • It follows a proven MVC architecture pattern (Beware, though, that there are some differences between the Rails-style MVC and the MVC implementation in Javascript frameworks).
  • Like Rails, it is opinionated software, i.e., it aims to relieve the developer from making unnecessary decisions concerning common and trivial implementation choices.
  • There are gems, like ember-rails, that facilitate the application of Ember.js in Rails 3.1+ applications. A Ruby on Rails developer is for sure familiar with the generators concept. With ember-rails you can generate full stack resources, wired with your Ruby on Rails models; you just need to use the generators ember-rails ships with. This can give a tremendous boost to productivity.
  • The Ember.js components like Ember Data provide many of the facilities you can find in server-side ORMs like ActiveRecord. The components are designed specifically for the JavaScript browser environment. Some features, e.g. the out-of-the-box support for Rails apps that follow the active_model_serializers gemâs conventions are still on the roadmap, but you can already get a sense of a general direction in which they are likely to evolve.
  • Ruby has had its impact on the Ember.js code and architecture (to name a few examples: mixins, super, namespaces). Multiple similarities between Rails and Ember.js, may allow Ruby on Rails developers to master the technology quite easily.

Let us look at the sample application we can find in the TodoMVC project repository. The application consists of 172 lines of code in the app.js file and the minified and packed Ember.js in the lib directory. Unlike with Backbone.js, no boilerplate adapters are needed. The code is cleaner than in the Backbone.js version. Still it could be even more pleasing to the eye, if it was written in CoffeeScript, like the Batman.js version. As you can see the computed properties allow you to treat a function definition as a regular property. What is more, the computed properties can have dependencies on other properties and if you list the dependencies properly, Ember.js makes all the binding updates when necessary. Neat!

Ember.js pros and cons

Pros:

  • Support for organizing views in a hierarchical order. Ember.View also allows you to encapsulate HTML templates, which enables you to use views composition techniques in your application (reusing templates).
  • A big part of Emberâs appeal comes from leveraging Handlebars.js declarative templates. After setting up the DOM (using helper methods like {{view}}, {{action}} and {{bindAttr}}), you no longer have to worry about keeping the DOM up to date.
  • Ember.jsoffers a lot of support for integration with Ruby on Rails and boosts developers productivity.
  • Very powerful implementation of bindings and computed properties.
  • Due to Ember.js design, it is easy to mix Ember.js with other Javascript libraries.

Cons:

  • Ember.js still exhibits limited adoption. Yet the project website features some respectable brand names, such as Zendesk or Square.
  • The Ember.js framework is in a state of flux as the framework has yet to hit 1.0. A lot of framework elements are changing and some of its parts are definitely not production ready, like the Ember Data module.
  • If I had written this article a few months ago, I would, without doubt, have added âPoor documentationâ in the cons section. But Ember.js maintainers have recently refreshed the project website with some nice docs and guides. For all the effort, you can still find some undocumented, quirky parts in the Ember.js framework.
  • Emberâs view binding system implementation can sometimes be the cause of poor performance of a single page application.
  • Ember.js loses against batman.js and Backbone.js when it comes to size: 42k min+gzip.
