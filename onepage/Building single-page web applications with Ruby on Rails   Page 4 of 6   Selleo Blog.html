Building single-page web applications with Ruby on Rails

Jun 28, 2012 by Radosław Jędryszczak    Posted under: Technologies

Page:← 1 2 3 4 5 6→

Backbone.js

Backbone.js was written in Javascript by Jeremy Ashkenas in 2010. There are several reasons why Backbone.js found its way into my listing of JS frameworks for building single-page applications with Ruby on Rails. The most important reason is that Backbone.js is optimized for the integration with RESTful back-ends, like RoR.

Web developers familiar with SPAs are highly likely to know Backbone.js. The framework has a strong community and a significant track record of usage in production applications. It has been used by 37signals, the company with significant contributions to Ruby on Rails. Backbone.js has also been used by LinkedIn, where Backbone.js powers their HTML5 mobile web application. Yet Backbone.js has a different philosophy than, for example, Ember.js, Ruby on Rails or other popular frameworks.

Rails is known for following the “Convention over Configuration” software design paradigm, which aims to decrease the number of decisions that RoR developers needs to make while writing software. CoC is not limited to proposing a standard set of configuration options for databases, server or the templating system; CoC includes abstracting patterns that are often used by coders. Hence, you may have heard about doing things “the Rails way”, which broadly means following some good practices to become more efficient at what you do.

Backbone.js takes a somewhat different approach. Rather than imposing a way of doing something, Backbone.js limits itself to defining the structure of a web application and introducing the concepts of views, models, a restful sync interface and routers. But at the same time, it is up to the developer to decide on how to architect the interaction between the aforementioned models and view, for instance. Such an approach has its advantages and limitations. This approach is, generally, good news for Rails developers, who prefer not to commit themselves to a single “full stack” solution, and bad news for those developers who prefer to learn a prefered way of doing something and benefit from access to powerful features.

I find it quite odd that despite all the minimalist approach of Backbone.js (leaving a lot of decisions to the developer), Backbone.js has so strict dependencies when it comes to the client-side scripting library. Backbone.js requires you to either use jQuery ( > 1.4.2) or Zepto.

This is not necessarily bad news for a Ruby on Rails developer, because since the release of Rails 3.1, jQuery has been the default JavaScript library for the framework. Another point for Backbone.js

Comparing Backbone.js to Batman.js or Ember.js, is similar to comparing Ruby on Rails and Sinatra. My colleague noted: “When I start building a new application in Ruby on Rails, I am often annoyed by the amount of bloat I introduce just by running the “rails new” command. But, when I start a project using Sinatra, I quickly find out I recreate the features I know from Rails”.

With expertise in Ruby on Rails at hand (which you can use to quickly build a powerful back-end for you single-page application), your choice boils down to whether you want to start developing the client from the Backbone.js minimalist point of view (aka Sinatra style) or, alternatively, approach client development from a “fully featured” Ember.js angle, in a Rails way.

OK, enough of this “Ruby on Rails vs. Sinatra” and “Backbone.js vs Ember.js” mumbo-jumbo. Let’s have a look at the code – the TodoMVC example provides something to look at: ~250 lines of code and the local storage adapter code (almost 100 lines). It is a lot when compared to the Batman.js example with the mere 200 lines in the JS-converted version. The amount of boilerplate code for such a simple application is rather significant. Thus I feel inclined to look for such solutions as the backbone-boilerplate to reduce typing and boost my productivity. Another thing worth noting is the way the code is structured – it is just not ‘pretty’. And rewriting it into CoffeeScript does not help much. I found out that it is easier for me to let my browser find phrases like “Backbone.Model“ and “Backbone.View” rather than do it myself. This issue raises concerns about the code structure if the application / codebase grows bigger.  Tutorials like this one explain how Backbone.js developers deal with the problem. Still I keep asking why we should “leave the developer in the dark” in the first place.
Backbone.js pros and cons

Pros:

  • Backbone.js works well with other JS frameworks and libraries.
  • The solution is designed towards consuming REST data; it works well with back-ends like Ruby on Rails.
  • Small size. The packaged and minified version of the library takes only 5.6KB.
  • Very good performance.
  • The minimalistic approach to the framework’s feature set ensures flexibility.
  • Backbone.js uses jQuery – the JS library every Ruby on Rails developer should be familiar with.
  • Composable and reusable views.

Cons:

  • Lack of support for nesting views inside one another, which is an important feature for building large SPAs and complex UIs.
  • Backbone.js is not framework agnostic like Batman.js, Backbone.js and requires either jQuery ( > 1.4.2) or Zepto. What can be viewed an advantage at first may turn out to be a limitation later on in the project.
  • Backbone.js does not support full customization regarding the CRUD pattern, i.e., Backbone lets you specify the response format of a Read, but, unfortunately, it does not allow you to fully customize rest serialization (Create, Update, Delete) when it comes to calling the server-side API.
  • A somewhat steep learning curve compared to some other JS frameworks like Knockout.js.
  • Less suitable for building large and complex single-page applications.
