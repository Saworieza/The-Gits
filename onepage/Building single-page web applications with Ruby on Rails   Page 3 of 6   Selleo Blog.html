Batman.js

According to the creators, Ruby on Rails developers should feel at home with Batman.js. The reason for this might be the fact that Batman.js was extracted and shaped into a framework from an existing Ruby on Rails application: Shopify. Batman.js is a framework for building rich single-page browser applications. It is written in CoffeeScript and its API is developed with CoffeeScript in mind. Batman.js is suitable for running both in a browser and in node.js ( v0.4.x and v0.6.x). As for the browser compatibility, Batman.js developers are targeting Chrome, Safari 4+, Firefox 3+, and IE 7+.
The code samples from the project site should illustrate why a Ruby on Rails developer may easily fall in love with Batman.js.

class Shopify extends Batman.App
 @root 'products#index'
 @resources 'products'

class Shopify.Product extends Batman.Model
 @persist Batman.RestStorage

class Shopify.ProductsController extends Batman.Controller
 index: ->
   @redirect action: 'show', id: 1

 show: (params) ->
   @product = Shopify.Product.find params.id

I do feel at home. The feeling may very well be aroused by Coffeescript leveraging, which was inspired by the Ruby language. As regards the implementation of the Batman.js sample application in TodoMVC project, it is hard to decide what to choose: the original coffeescript file or the javascript conversion. The former reflects the spirit of the framework better and lends itself better to present the features. The js-converted version ensures consistency â  all other examples are written in Javascript. However, the automatically converted code looks just ugly and is very hard to read. Since I consider using Coffeescript an advantage, I eventually decided to use the Coffeescript version.

The Todo app code is very well organized and the structure resembles the architecture of an RoR application: there are routes, model and controller sections. Below, you will find a Batman.js model example from the TodoMVC project. It is a Todo application class with a Rails-style looking definitions concerning encoding, persisting, validations and storageKey, as well as accessors definitions:

class Alfred.Todo extends Batman.Model

@encode 'title', 'completed'
@persist Batman.LocalStorage
@validate 'title', presence: true
@storageKey: 'todos-batman'

@classAccessor 'active', ->
@get('all').filter (todo) -> !todo.get('completed')

@classAccessor 'completed', ->
@get('all').filter (todo) -> todo.get('completed')

@wrapAccessor 'title', (core) ->
set: (key, value) -> core.set.call(@, key, value?.trim())

Batman.js pros and cons

Pros:

  • A routable controller actions
  • Valid HTML5 templates and a powerful binding system.
  • Batman.js is a JS framework agnostic â it does not force you to use a specific solution like  jQuery or Prototype.
  • Clean codebase written in CoffeeScript!
  • Batman.js authors focused on developersâ happiness and productivity. All I can say, is Batman.js gets it.
  • A very small size of the library: only 15KB gzipped.

Cons:

  • Incomplete documentation. Currently, only about 50% of the Batman.js framework is documented. That is a serious flaw.
  • Batman.js lacks composable view templates / handlebars that can be found, for example, in Ember.js. Composable view templates allow the developer to take a view-first approach to the composition of widgets (views) displayed on the page. In this technique views can be reused or adapted for different scenarios. Handlebars allow you to configure a view template association with a specific controller. Hence, the views can be composed at the template level and the developer does not have to worry about the view-controller communication.
  • A built-in support (in batman-rails, for example) for compiling / minifying views into packages would be handy. I consider sending each view separately for each request a less superior technique than just sending all the views packaged into one file, in one request. Fortunately, the problem can be solved; there is a good tutorial how to leverage Rails 3.1+ asset pipeline to create a json file that contains all of our appâs views.
